### 1.4 Codex Desktop App：与 Claude Code 的功能对比 🡒

- OpenAI 的界面“明显高于 Anthropic”——插件很直观，Claude 的 Connectors 让人困惑
- Codex 限制 6 个子智能体；Claude Code 支持最多 16 个
- Codex 子智能体必须显式调用；Claude Code 会自动调用
- Codex 插件不能包含子智能体
- 使用 gpt-5.5 xhigh 的 fast mode，在每天 14 小时工作会话中 2.5 天烧完每周额度
- 自动审查权限模式“效果相当不错”——比 Claude Code 的 auto mode 更好
- 云端智能体限制在 gpt-5.3-codex，且没有基础设施即代码环境设置

## 3. 人们期望的功能

### Token 成本可见性和优化成为默认能力

[@InduTripat82427](https://x.com/InduTripat82427) [整理](https://x.com/InduTripat82427/status/2049894134417326432)10 个降低 token 消耗的仓库（RTK: 60-90%，Context Mode: 98%，Token Savior: 97%）。开发者需要自己发现、安装并叠加 2-3 个独立工具来管理 token 消耗，说明这应该内建在智能体本身。没有编程智能体提供原生 token 预算管理或任务前成本预估。

紧迫性：高。机会：直接 —— GitHub Copilot 按用量计费将于 6 月 1 日生效。

### Codex 子智能体与 Claude Code 看齐

[@aniketapanjwani](https://x.com/aniketapanjwani) [指出](https://x.com/aniketapanjwani/status/2049661842373451855)三个缺口：Codex 限制 6 个子智能体（Claude Code: 16）、子智能体需要用户显式调用（不会自动生成）、插件不能包含子智能体。对能从并行分析中获益的代码审查工作流而言，6 个智能体上限是主要瓶颈。

紧迫性：中。机会：直接 —— 这由 OpenAI 控制；高阶用户反馈可能加速改进。

### 面向智能体工作流的 worktree 环境引导

[@aniketapanjwani](https://x.com/aniketapanjwani) [指出](https://x.com/aniketapanjwani/status/2049661842373451855)：“如果 Codex 能内置一些用于引导 worktree 环境的技能就好了——类似 Claude Code 用 /update-config 来初始化仓库权限。”他构建了自己的 `/worktree-cli-bootstrap` 技能，但认为这种摩擦阻碍了 worktree 的采用。

紧迫性：中。机会：间接 —— 社区技能可以在官方支持前解决。

## 7. 机会在哪里

**[+++] 作为原生智能体功能的 token 预算管理** -- [@InduTripat82427](https://x.com/InduTripat82427) [整理](https://x.com/InduTripat82427/status/2049894134417326432)的 10 个降低 token 消耗的仓库（RTK：终端输出省 80%，Context Mode：SQLite 沙箱隔离省 98%，Token Savior：导航省 97%）说明，token 管理目前分散在社区工具里。GitHub Copilot 按用量计费将于 6 月 1 日开始，Codex 高阶用户也会在 2.5 天内烧完每周额度（aniketapanjwani），因此原生 token 预算管理——任务前预估成本、按分支归因、自动优化——是最直接的机会。6 月 1 日后，每个智能体用户都可能成为客户。

**[+++] 智能体技能市场与发现** -- Obsidian Skills（几天内 27K 星标）、andrej-karpathy-skills（总计 101K、本周 24K）、Matt Pocock 的工程技能，以及 awesome-copilot 仓库，都显示对组合式智能体技能的爆发需求。还没有统一市场用于跨 Claude Code、Codex、OpenCode 发现、评分和组合技能。`npx skills add` 模式已经存在，但发现仍靠 Twitter 口碑。

**[++] 面向智能体式编程的视觉工作空间** -- [@nimbalyst](https://x.com/nimbalyst) 的 [开源发布](https://x.com/nimbalyst/status/2049849943133749541)为 Codex/Claude Code/OpenCode 提供 Markdown、产品草图、架构图和代码差异的可视化编辑器，回应了终端智能体缺少视觉反馈的持续抱怨。Codex Desktop App UI 更好（aniketapanjwani）与 Claude Code 纯终端界面之间，留下了一个可为任何智能体添加视觉层的工具缺口。

**[+] 提供商无关的运行框架基准测试** -- [@caspar_br](https://x.com/caspar_br) 的 [deepagents profiles](https://x.com/caspar_br/status/2049672406260924729)可按提供商调优运行框架，带来 10-20 分改进。大多数开发者仍使用默认配置。一个能在你的具体代码库上对多个运行框架 Profiles 做基准测试、并推荐最优设置的工具，可以捕获目前需要手动实验才能获得的价值。

## 8. 要点总结

1. **Microsoft 的 AI 经济学面临公开信誉挑战。** [@edzitron](https://x.com/edzitron) 的 [批评](https://x.com/edzitron/status/2049885224738472366)（204 点赞，11,543 浏览量）把叙事从“Copilot 定价变化”改写为“$300B 投资是否曾经可行”。外界把转向按 token 计费看作单位经济模型崩坏的承认，而不是常规调价。这给了每个反 Microsoft 锁定的替代品弹药。

2. **Codex 正从编程工具转向通用工作平台。** [角色选择器](https://x.com/aakashgupta/status/2049966530822332502)（Engineering 到 Student）、[CLI 0.128.0](https://x.com/fcoury/status/2049917871799636201) 中的 /goal 持久化，以及 GPT-5.5 发布派对，都说明 OpenAI 正把 Codex 构建成横向平台。[@blakeandersonw](https://x.com/blakeandersonw) 的 [反驳](https://x.com/blakeandersonw/status/2049975392841158938)称 Claude 在“二三线知识工作者”中动能强得多，说明竞争目标是非开发者多数。

3. **智能体技能是新的价值创造层。** Obsidian Skills（27K 星标）、andrej-karpathy-skills（101K 星标，24K/周）、free-claude-code（85% 周增长）和 Matt Pocock 的工程技能表明，真正的差异化不再是你用哪个模型，而是你挂了哪些技能。token 优化仓库（RTK、Context Mode）说明技能正在解决智能体自身没有解决的问题。

4. **运行框架工程正在产品化。** [@caspar_br](https://x.com/caspar_br) 的 [deepagents profiles](https://x.com/caspar_br/status/2049672406260924729)（10-20 分提升）和 Cursor 的按模型调优运行框架（Quan2m2）把运行框架优化从黑客文化（4 月 29 日的 goblin mode）带进主流工具。这个认识已经成为共识：同一模型 + 更好的运行框架 = 明显更好的输出。

5. **Antigravity 的可靠性侵蚀进入第六天。** [@amritwt](https://x.com/amritwt)（161 点赞）、[@TheGlobalMinima](https://x.com/TheGlobalMinima)、[@y_qecea](https://x.com/y_qecea)（14,127 浏览量）和缺失 Opus 4.7 模型（luicho9_）延续了此前模式。$20/月价值主张（immasiddx，32 点赞）仍是唯一辩护，但前提是服务真的可用。

6. **Codex vs Claude Code 的对比现在是功能级，而不是模型级。** [@aniketapanjwani](https://x.com/aniketapanjwani) 的 [详细对比](https://x.com/aniketapanjwani/status/2049661842373451855)（29 收藏数）——子智能体上限、权限模式、插件架构、环境引导——说明竞争已经从“哪个模型代码写得更好”转向“哪个运行框架更能处理复杂工作流”。下一波差异化会发生在这里。
