### 1.5 对抗式多模型工作流成为实践 🡕

[@0xLatencyZero](https://x.com/0xLatencyZero) [发布工作流图](https://x.com/0xLatencyZero/status/2047739858656067709)（1 点赞，137 浏览量，2 收藏），介绍“对抗式审查”——用 Claude Code 中的 Opus 4.7 起草代码，再用 Codex 插件中的 GPT-5.5 审查 diff：“两款前沿模型。不同训练背景、不同盲点、独立审查。” 流程是：开发者把意图发送给 Claude Code，后者起草并构建；diff 进入 Codex，由它标记逻辑漏洞、回归和安全漂移；审查后的代码再发出。

![对抗式审查工作流显示 Claude Code Opus 4.7 起草代码，然后 Codex GPT-5.5 审查 diff，查找逻辑漏洞和回归](https://pbs.twimg.com/media/HGsHxzQbwAASSQR.png)

[@mr_the_dooom](https://x.com/mr_the_dooom) [描述类似模式](https://x.com/mr_the_dooom/status/2047561045041156219)（0 点赞，27 浏览量）：“把 Claude 纯粹当成经理、架构师和文档维护者，把 codex 当成由 Claude 驱动的写代码苦工。不知怎么地，这种搭配产出惊人，但 token 用量却很少。”

Nadella 的讨论串在平台层面强化了这一点，描述了 “Rubber Duck 智能体”，其中 “GPT-5.5 可以审查另一个模型的输出，反之亦然。”

**讨论要点：** 对抗式多模型审查正同时以自上而下（Nadella 将其描述为平台功能）和自下而上（实践者独立发现）的方式出现。共同洞察是：来自不同实验室的模型有不同失败模式，因此跨实验室审查能抓住同模型审查漏掉的缺陷。

**与前日对比：** 4 月 23 日，测试框架质量被指出比模型选择更重要（kylejeong 展示同一模型不同输出）。今天模式延伸到有意设计的多模型流水线：不同实验室的模型承担互补角色，一个起草，另一个审查。

### 1.7 Copilot SDK v0.3.0 与 JetBrains Agent Mode 发布 🡕

1. **GPT-5.5 采用正在推动真实订阅变化，而不只是基准测试热情。** [@mckaywrigley](https://x.com/mckaywrigley) [报告](https://x.com/mckaywrigley/status/2047806060061507971)，不到三个月内从 80/20 Claude/GPT 翻转到 80/20 GPT/Claude。[@Kappaemme1926](https://x.com/Kappaemme1926) [发布 Claude 取消订阅邮件](https://x.com/Kappaemme1926/status/2047598921996984495)。[@Colewherld](https://x.com/Colewherld) [称 Codex](https://x.com/Colewherld/status/2047684720675221653) “比 Claude Code 强太多了。” 发布后第二天的信号是迁移，而不只是评估。

2. **限流是每个提供商的核心用户体验问题。** [@notjazii](https://x.com/notjazii) [在 36 小时内烧穿](https://x.com/notjazii/status/2047616448818708828) Antigravity 配额（90 回复，数据集最高）。[@MilkRoadAI](https://x.com/MilkRoadAI) [报道](https://x.com/MilkRoadAI/status/2047797528293188024) Anthropic 下调 Claude 思考深度。2026 年 $700B 的超大规模云厂商合计资本开支仍无法满足需求。算力短缺是结构性的，不是暂时性的。
